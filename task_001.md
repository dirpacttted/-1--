Собственно сама задача:
    Разработать приложение которое сможет сравнить несколько конфигураций 1С (конфигурации хранятся в виде набора файлов формата EDT) в части установки признаков разделения объектов метаданных для корректной работы в 1С:Фреш.

Так как задача достаточно объемная разделим ее на несколько частей:
1.	Разработка каркаса приложения и структуры хранения данных.
2.	Разработка механизма парсинга конфигурации 1С из каталога и сохранение данных в базе.
3.	Реализация механизма вывода данных загруженной конфигурации 1С в интерфейсе в виде дерева (по аналогии с деревом метаданных 1С) и отображение для объектов метаданных конфигурации режимов разделения.
4.	Реализация механизма сравнения двух и более загруженных конфигураций в виде общего дерева и визуализация расхождения режимов разделения для пересекающихся объектов метаданных.
 
Все настройки и данные приложения хранятся во внешней базе данных PostgreSQL либо MySQL. Соответственно на первом этапе нам нужно:
1.	Реализовать при запуске приложения чтение конфигурационного файла, в котором хранятся параметры подключения к серверу СУБД.
2.	Проработать структуру таблиц для хранения в СУБД конфигурационной информации, а именно:
2.	
    a.	нам нужно хранить данные по самой конфигурации: Имя, Синоним, путь к каталогу загрузки по умолчанию.
    b.	нам нужно для конфигураций предусмотреть возможность хранения разных версий.
    c.	в разрезе версии конфигурации мы должны хранить объекты метаданных, возможно для них стоит отдельно хранить как минимум тип, имя, синоним, уникальный идентификатор объекта. И для хранения данных разделения нам нужно сохранить для каждого из объектов значение общего реквизита "ОбластьДанныхВспомогательныеДанные" и "ОбластьДанныхОсновныеДанные", но нужно помнить, что для некоторых объектов метаданных разделение не предусмотрено.
3.	Результатом должна быть схема таблиц с описанием полей и типов данных, описанием ключей и связей между таблицами
4.	По результатам разработки структуры необходимо предусмотреть в приложении механизм проверки структуры базы данных, к которой произошло подключение. Если структура не соответствует эталонной или вообще отсутствует ее необходимо создать/исправить для соответствия эталонному виду.
