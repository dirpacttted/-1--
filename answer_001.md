# Проверка выполнения задачи 1 этапа (`task_001.md`)

## Итоговая оценка: **7 / 10**

Каркас приложения и структура хранения данных в целом соответствуют постановке первого этапа: есть чтение конфигурации, схема БД под конфигурации/версии/метаданные/разделение и механизм создания таблиц при их отсутствии. При этом проверка структуры реализована поверхностно (только наличие таблиц, без сверки и исправления полей/ключей), в коде и зависимостях есть заметные дефекты качества.

## Замечания (критичные и существенные)

### 1. Проверка структуры БД не соответствует формулировке задания

В `dbmanager.check_database_structure()` проверяется только факт наличия имён таблиц. Не проверяются:

- состав и типы столбцов;
- первичные/внешние ключи;
- ограничения уникальности.

Если таблицы существуют, но схема «сломанная» или устаревшая, приложение выведет «Структура базы данных соответствует.» и **не исправит** её. Задание явно требует: при несоответствии эталону структуру **создать/исправить**.

Также при частичном наборе таблиц (есть 2 из 4) срабатывает ветка создания: `CREATE IF NOT EXISTS` досоздаст недостающие, но уже существующие неверные таблицы не тронет.

**Рекомендация:** сверять эталон (из `schema.sql` или отдельного описания метаданных схемы) с `INFORMATION_SCHEMA`; при расхождении — миграции / `ALTER TABLE`, либо контролируемый пересоздание в dev-режиме. Минимум — проверять обязательные столбцы и FK.

### 2. Несогласованность имён таблиц (регистр)

В `schema.sql` таблицы создаются в PascalCase (`Configurations`, `ConfigurationVersions`, …), а в коде проверяются имена в нижнем регистре:

```python
required_tables = [
    "configurations",
    "configurationversions",
    ...
]
```

На Windows / при `lower_case_table_names=1` это часто «сходит с рук», на Linux (типичный Docker) регистр может отличаться — проверка начнёт ложно срабатывать или, наоборот, пропускать таблицы. Нужно зафиксировать единый стиль имён (предпочтительно `snake_case`: `configurations`, `configuration_versions`, …) и использовать его и в SQL, и в Python.

### 3. Отсутствуют уникальные ограничения, важные для предметной области

Сейчас возможны дубликаты:

- несколько конфигураций с одним `name`;
- несколько записей одной версии у одной конфигурации (`configuration_id` + `version`);
- дубли объектов метаданных в рамках версии, например в конфигурации может сущестововать одноименная константа и функциональная опция, это тоже нужно учитывать;
- несколько строк разделения на один объект (в отчёте заявлено 1:1, в схеме UNIQUE нет).

**Рекомендация:**

```sql
UNIQUE (name)  -- Configurations
UNIQUE (configuration_id, version)  -- ConfigurationVersions
UNIQUE (version_id, object_uuid)  -- или (version_id, object_type, name)
UNIQUE (metadata_object_id)  -- DataSeparationSettings
```

### 4. Учёт объектов без разделения описан слабо

Задание напоминает, что для части объектов разделение не предусмотрено. Отдельная таблица `DataSeparationSettings` это допускает (нет строки = нет разделения), но:

- это нигде явно не зафиксировано в коде/документации схемы как правило предметной области;
- поля `auxiliary_data_area` / `main_data_area` допускают NULL даже при наличии строки — семантика «не применимо» vs «не задано» не разделена.

**Рекомендация:** в описании схемы явно указать: отсутствие записи в `DataSeparationSettings` означает, что разделение для объекта не предусмотрено; при наличии записи значения обязательны (или ввести флаг / справочник режимов).

### 5. `requirements.txt` содержит мусор

```
dbmanager
schema
config
PyYAML
mysql-connector-python
```

Строки `dbmanager`, `schema`, `config` — не пакеты PyPI. Установка зависимостей (`pip install -r requirements.txt`) будет падать или вести себя непредсказуемо.

**Рекомендация:** оставить только реальные зависимости, желательно с версиями:

```
PyYAML>=6.0
mysql-connector-python>=8.0
```

### 6. Учётные данные БД в репозитории

В `config.yaml` и `docker-compose.yml` зашит пароль (`root097`). Для учебной работы допустимо локально, но в git лучше:

- `config.example.yaml` без секретов;
- рабочий `config.yaml` в `.gitignore`;
- либо переменные окружения.

### 7. Нет обработки ошибок

При отсутствии `config.yaml`, неверном YAML, недоступной СУБД, отсутствии БД `config_storage` приложение завершится необработанным исключением. Для каркаса стоит хотя бы:

- понятные сообщения пользователю;
- `try/finally` или context manager для закрытия соединения;
- проверка наличия ключей в конфиге.

## Замечания по качеству кода и best practices

1. **Смешение отступов в `main.py`** (табы и лишний уровень в `load_config`) — ухудшает читаемость, может ломать стиль-чекеры.
2. **Жёсткие относительные пути** (`"config.yaml"`, `"schema.sql"`) зависят от текущей рабочей директории. Надёжнее строить путь от расположения скрипта (`pathlib.Path(__file__).resolve().parent`).
3. **Разбор SQL через `split(";")`** хрупок (комментарии, процедуры, `;` внутри строк). Для учебного DDL приемлемо; в перспективе — выполнять скрипт пошагово или через миграции (Alembic и т.п.).
4. **Нет type hints и docstrings** у публичных методов `DatabaseManager` — для учебного каркаса желательны.
5. **README** почти пустой: нет инструкции по запуску Docker, установке зависимостей и старту `main.py`.
6. **Индексы по FK** не созданы явно (MySQL часто индексирует FK сам, но лучше не полагаться молча и документировать).
7. Типы значений разделения (`НеИспользовать`, `Независимо`, …) логичнее ограничить справочником/ENUM или CHECK, а не свободным `VARCHAR(255)` — на следующих этапах это упростит сравнение.

## Что сделано хорошо

- Логичная нормализованная схема: конфигурация → версии → объекты метаданных → настройки разделения.
- Каскадное удаление по FK (`ON DELETE CASCADE`) уместно для связки версия/объекты.
- Вынесение схемы в отдельный `schema.sql` и текстовое описание полей/связей.
- Наличие `docker-compose.yml` упрощает локальный стенд MySQL.
- Разделение ответственности: `main.py` (запуск) / `dbmanager.py` (работа с БД).
- Использование `yaml.safe_load` — правильный выбор.
